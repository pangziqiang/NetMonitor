📊 网络监控小结

NetMonitor 在网络流量深度分析上全面领先。 其 SQLite 持久化架构支持任意时间段的流量回溯、进程级钻取、数据导出——这是 Stats 完全不具备的能力。 但 Stats 在网络连接的辅助信息（IP/SSID/延迟/连通性检测）和接口管理上更完善。

9. 总体评价与建议

🏆 定位完全不同——互补而非替代

Stats 是一个通用系统仪表盘，用 11 个模块覆盖几乎所有硬件传感器， 44 种语言、14 种小部件类型，是 macOS 上最流行的开源系统监控工具。适合想一眼看到 CPU/内存/磁盘/电池/温度的普通用户。

NetMonitor 是一个网络流量分析工具，其 SQLite 持久化架构和三级粒度（分钟/小时/天） 历史数据在 macOS 生态中几乎没有竞品。双击钻取进程级流量的交互设计非常实用。 适合需要追踪流量去向、做对账分析、排查"哪个应用偷跑了流量"的重度用户。

NetMonitor 应优先补强的领域

#改进方向参考 Stats优先级
1 磁盘监控模块 — 添加磁盘利用率、读写速度 Disk.framework 高
2 通知系统 — CPU/内存/流量阈值告警 Notifications 高
3 开机启动 — Login Item 注册 Start at Login 高
4 网络接口 GUI 选择 — 替代手动前缀配置 Network interface picker 中
5 公网 IP / WiFi 信息 — 显示 SSID、IP、延迟 Network IP/WiFi 中
6 数据导出增强 — 支持 CSV/JSON Schema 选择 — 中
7 多语言本地化 — 补全 L10n 键、增加英文 44 languages 低
8 自动更新机制 — Sparkle 或自建更新 Auto update 低
9 电池监控模块 — 电量、健康度、循环数 Battery.framework 低

Stats 缺乏但 NetMonitor 独有的亮点

#能力价值
1 分钟/小时/天三级流量历史数据库 可回溯 730 天任意日期的流量数据，用于对账、审计、趋势分析
2 进程级流量历史 + 双击钻取 点击任意小时柱子即可看到该时段哪些应用用了多少流量——这在排障时极有价值
3 流量数据导出 诊断 JSON 导出，可做进一步分析或对账
4 浮动窗口 独立于菜单栏的桌面悬浮窗，始终可见
5 Pin 固定弹窗 弹出窗口可固定不关闭，方便对比观察

结论：两者是互补关系，不是替代关系。

Stats 是"瑞士军刀"——什么都能看，但什么都不深。NetMonitor 是"手术刀"——只做网络流量一件事，但做到极致。

如果你想监控 CPU/GPU/内存/磁盘/电池/温度/风扇，用 Stats。
如果你想追踪每 MB 流量去了哪里、哪个应用偷跑了流量、过去一个月流量趋势如何，用 NetMonitor。

理想组合：两者同时运行，各取所长。

结论 每次采样做 5 项额外工作
→ CPU 峰值 14-17%		 每次采样只做 1 项工作
→ CPU 峰值 5-6%
根因不是代码质量问题，而是功能代价。

Stats 是纯展示工具——读取传感器 → 更新 UI → 完毕。
NetMonitor 每 15 秒执行：
  ① JSON 序列化进程数据
  ② INSERT 分钟流量记录
  ③ INSERT/UPDATE 天级汇总
  ④ INSERT 进程流量明细
  ⑤ 小时级数据回填聚合 (GROUP BY + INSERT OR REPLACE)
  ⑥ WAL 日志写入磁盘

这些操作是 11MB 数据库、三级历史粒度、进程级流量追溯 必须付出的代价。
CPU 峰值 14–17% 持续时间 < 200ms，每 15 秒一次，不影响系统流畅度。

如果关闭数据持久化，NetMonitor 的 CPU 基线将与 Stats 持平。

A.6 运行时综合评价

内存：势均力敌 (~117 MB RSS)，NetMonitor 线程更少 (9 vs 16)。

CPU：NetMonitor 峰值更高但原因明确——SQLite 持久化是主动选择的功能而非性能缺陷。关闭持久化则 CPU 基线持平。

稳定性：均稳定无崩溃，无内存泄漏迹象。

结论：Stats 适合"常驻菜单栏极低功耗"场景，NetMonitor 以可接受的 CPU 代价换取了业界独一档的历史数据分析能力。
